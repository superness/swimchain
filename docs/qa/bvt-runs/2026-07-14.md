**CORRECTION — M4 is NOT a launch blocker (I was wrong; recorded so it doesn't
recur).** Names ARE on-chain content and DO propagate: a space whose name was
committed on-chain (space_metadata on its CreateSpace) resolves on a fresh node
— verified, qa2 synced "Latency Lab 2" correctly. The design is on-demand:
clients request a space's content (name included) when viewing it; nodes do not
and should not eagerly gather everything. The repair path (resolve_space_name →
GET_SPACE_META → handle_space_meta upsert) exists and works.

What shows raw `sp1…` on qa2 is reef/chess/drift, whose names live ONLY in the
seed/genesis config.toml operator override (`list_spaces` merges it at list
time) and were never written on-chain — so there is nothing for any node to
request. The seed's own resolve_space_name returns null for The Reef, proving
it's config-only. Operator setup artifact, not a protocol gap; users reach the
games via clients that carry app-names on-chain anyway. Optional cleanup (not a
blocker): commit those names on-chain (named CreateSpace/rename) instead of
config.toml, or treat config names as operator-display-only.
