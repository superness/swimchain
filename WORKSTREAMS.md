# Swimchain - Workstreams for Resolving Open Questions

## Overview

The vision document contains many open questions. These require different resolution methods depending on their type. This document organizes them into workstreams with concrete deliverables.

---

## Workstream 1: Technical Prototype

**Goal:** Replace assumptions with measurements.

**Questions to Answer:**
- What chain size does decay produce at various scales (1K, 10K, 100K users)?
- What PoW duration on typical hardware (phone, laptop, desktop)?
- How long does initial sync take?
- Can background sync keep mobile nodes current?
- What bandwidth does participation require?

**Approach:**
1. Build minimal prototype with core mechanics only
2. Simulate usage patterns at various scales
3. Benchmark on real devices
4. Document findings

**Deliverables:**
- [x] Minimal prototype (chain + PoW + decay + sync) - Phases 0-4 complete
- [x] Simulation framework for scaling tests - tests/mobile_simulation/, tests/network/, tests/e2e_flows/
- [x] Benchmark results across device types - docs/benchmarks/mobile.md, docs/benchmarks/pow.md
- [x] Technical feasibility report with real numbers - docs/mobile-viability.md

## Workstream 4: Team Decisions ✅ COMPLETE

**Goal:** Make explicit value choices where reasonable people disagree.

**Status:** All four decisions resolved December 2024. See VISION.md "Team Decisions" section for full rationale.

### Decision: Target User ✅

**Resolution:** Technical users first, with path to broader adoption.
- Early adopters must be technical (keys, clients)
- UX investment at client layer
- ~10K committed users may be sufficient

### Decision: Minimum Viable Decentralization ✅

**Resolution:** Progressive trust with eventual verification.
- Users are ephemeral participants, not persistent nodes
- Trust temporarily, verify eventually
- Decay bounds chain → full verification realistic for all

**Key architectural insight:** Swimchain is a hybrid:
- **Authoritative layer** (Bitcoin-like): Chain of signed, PoW-verified post records
- **Content layer** (BitTorrent-like): Media blobs, content-addressed, seeded by interest

### Decision: Handling Illegal Content ✅

**Resolution:** Hash blocklists at protocol level.
- CSAM hash detection is non-negotiable (legal requirement)
- Nodes refuse to store/relay matching content
- Minimal centralization (hash database from NCMEC/IWF)

### Decision: Development Sustainability ✅

**Resolution:** Open source + grants, explicit rejection of foundation model.
- Foundation creates central target (Tornado Cash precedent)
- Volunteer open source + grants from EFF, Mozilla, crypto foundations
- Accept slower development for no central entity
